## Predict–Discover–Explain checkpoints

### 1. Graph storage

**Predict:** Does this GGUF contain an executable GGML graph?

**Discover:** Run the parser and inspect the sections it reports.

<details>
<summary>Explain</summary>

No. The bounded fixture contains a GGUF header, metadata, tensor descriptors, alignment padding, and tensor payload bytes. Native llama.cpp constructs a GGML graph later; this browser lab does not execute that path.

</details>

### 2. Tensor offsets

**Predict:** Is `demo.vector` stored immediately after `demo.matrix`?

**Discover:** Compare the end of `demo.matrix` with the absolute start of `demo.vector`.

<details>
<summary>Explain</summary>

No. `demo.matrix` occupies bytes 384–399. `demo.vector` starts at byte 416 because its relative offset is aligned to 32 bytes, leaving bytes 400–415 as padding.

</details>

### 3. Mapping versus residency

**Predict:** Does parsing a descriptor—or mapping a file—prove every tensor page is physically resident?

**Discover:** Identify what this browser parser actually observes.

<details>
<summary>Explain</summary>

No. This parser reads a 428-byte in-memory teaching fixture and reports file structure. It does not invoke native `mmap`, inspect page tables, count page faults, or prove physical residency.

</details>
