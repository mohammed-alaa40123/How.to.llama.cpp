1. GGUF bytes on storage

The file contains metadata plus typed tensor payloads. Loading may use direct reads or an mmap-backed path.

2. Virtual mappings

With mmap, model tensor addresses refer into a file-backed virtual mapping. Addressability does not prove physical residency.

3. Page faults and RAM

First access can fault pages into the OS page cache. Clean file-backed pages may later be reclaimed without writing model data back.

4. Model-owned tensor metadata

ggml_tensor records shape, type, strides, operation metadata, and a data pointer or backend allocation relationship.

5. Context runtime memory

llama_context owns or references scheduler state, compute buffers, outputs, and sequence memory modules such as KV or recurrent state.

6. Backend copies and staging

Cross-backend edges may allocate destination copies. Unsupported direct paths can use synchronized host staging.

7. Temporary graph allocations

Graph tensors, views, copy-ring slots, and backend workspaces follow graph/scheduler lifetimes rather than model-file lifetime.

8. Teardown and reclaim

Context resources are released before model mappings. OS page-cache residency is independently managed by the kernel.

GGUF container

Verified: GGUF stores a header, versioned metadata key/value entries, tensor descriptors, alignment information, and tensor data. The loader uses metadata to identify architecture, hyperparameters, tokenizer/vocabulary information, and tensor layout.

Official GGUF specification

Tensor registration

Model loading creates GGML tensor metadata corresponding to named GGUF tensors and associates those tensors with mapped, read, or backend-managed storage.

Graph construction

Architecture-specific builders call GGML operations. Each operation returns a tensor whose op and source pointers describe one node of the lazy computation graph.

Graph expansion

Starting from output tensors, graph expansion walks dependencies and inserts operation tensors into ggml_cgraph in execution order.

Activations versus weights

Weights generally reference model-lifetime storage. Activations and intermediate tensors require graph/backend allocations and usually have shorter lifetimes.

Layer-by-layer execution

Interpretation: the graph is logically organized by transformer layers, but execution and memory reuse are controlled by graph topology, allocator lifetimes, backend splits, and backend kernels—not a universal “load one layer, free it immediately” rule.

Important correction

llama.cpp with an mmap-backed model can fault weight pages on demand and the OS may reclaim clean pages later. That is different from an explicit application-level policy that loads exactly one layer, executes it, and frees it after every token. Backend offload and scheduler copies introduce additional storage lifetimes.
